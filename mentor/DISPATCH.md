# 派单模型:看板 ↔ 执行器分离

这套系统刻意把两种角色拆开。任何会话打开本仓库,先判断自己是哪种角色。

## 两种角色

### 🗂 看板会话(Board)—— 只调度,不执行
职责:维护 `TODO.md` / Issue #1 / 作战板;每小时检查、记录、督促;把具体任务**派**给执行器。
**看板不亲自做具体任务**(不写 deck、不做研究、不起草长文档)——那会烧光它的上下文,而它的价值在于长期在线的调度。
唯一例外:几分钟内能完成的碎事(改日期、勾选、记录)。

### ⚙️ 执行器会话(Executor)—— 只执行,不留状态
职责:领一个具体任务,把它做完,产出落到仓库 `drafts/`,更新 TODO/Issue,推送。
执行器是**一次性**的——做完即弃,状态全部写回仓库。上下文烧完不心疼,因为记忆不在它那里。
每个执行器开工第一步:读 `CLAUDE.md` 了解规则,读 `TODO.md` 找到自己那条任务。

## 三条派单通道(按"零配置→需配置"排序)

| 通道 | 怎么触发 | 在哪执行 | 是否需配置 |
|------|---------|---------|-----------|
| **A. GitHub @claude** | 在 Issue #1 评论 `@claude <任务>` | GitHub Actions 里全新会话,开 PR | 装 github.com/apps/claude + `ANTHROPIC_API_KEY` secret(走 API 计费) |
| **B. 作战板按钮** | 作战板每个任务的 `▶ 派单` / `⧉ 复制` | 本机 Claude Code / 粘到 Cowork | ▶ 需本机装 Claude Code;⧉ 无需 |
| **C. 派单 Routine** | 看板会话里说"派单:<任务>" | 云端全新会话,推送结果 | 需一次连接器授权 |

## 派单提示词模板(所有通道通用)

```
你在仓库 AdenProSDE2024/claw-code-(规则见 CLAUDE.md)。执行任务:「<任务标题>」。
第一步:先把完成所需的 context 一次性问齐(背景、涉及的人、已有素材、成功标准、格式)。
第二步:据此执行/起草,文字成果放进 drafts/。
第三步:完成后在 TODO.md 标记 [x],运行 python3 mentor/tracker.py record,commit 并 push;
        在 Issue #1 对应任务下留言产出位置。
```

## 分工建议(什么派出去、什么自己做)

- **适合派给执行器**:起草类(邮件、proposal、协议、deck 大纲)、研究类(名单、尽调)、模板类(reach out 模板)——本会话已验证 deep-research、drafts 都能由执行器独立完成。
- **只能你本人做**:需要内部/私密 context 的(Monad deck 内容、和具体人的关系判断)、需要你决策拍板的。
- **看板(这个会话)保留**:调度、督促、记录、把你口述的新任务归档进 TODO/Issue。
